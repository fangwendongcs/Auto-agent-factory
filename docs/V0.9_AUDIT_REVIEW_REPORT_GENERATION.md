# V0.9 Audit Review Report Generation

Target:

```text
Generate a human-readable audit review report from sanitized audit records.
```

This phase reads sanitized audit records and renders a Markdown report. It does not modify n8n runtime, call providers, read credentials, or write production files.

## 1. What was implemented

Repo-side implementation:

```text
src/utils/generateAuditReviewReport.js
scripts/generate-audit-review-report.mjs
tests/audit-report.test.js
```

Package script:

```bash
npm run audit:report
```

## 2. Inputs

The report generator accepts:

- a single sanitized audit record JSON file
- a JSONL file containing sanitized audit records

Examples:

```bash
node scripts/generate-audit-review-report.mjs examples/audit-replay/v08f-read-only-run-record.json
node scripts/generate-audit-review-report.mjs .local-audit/v08f-replay.jsonl
```

The CLI validates each input record before rendering the report.

## 3. Output

The CLI writes Markdown to stdout by default.

V0.9b adds a disabled-by-default local artifact option. When explicitly enabled, it can write Markdown only under `.local-audit/reports/*.md`.

The report includes:

- run id / task id
- status
- risk level
- action class
- permission level
- approval decision
- provider mode / status
- storage mode / persisted flag
- no-write safety flags
- redaction status
- forbidden / high-risk / write-like flags
- safe-for-review status

## 4. Safety boundaries

The report generator must not require or display:

- full prompts
- provider messages
- provider raw full response
- Authorization headers
- API keys
- n8n API keys
- provider credentials
- cookies
- passwords
- tokens
- secrets

It only reads sanitized fields that already passed audit record validation.

## 5. Review semantics

A record is marked safe for review only when:

- no side effects are enabled
- raw prompts are not stored
- raw provider responses are not stored
- secrets are not stored
- redaction is applied
- no raw prompt/provider fields are present
- no secret-like values are detected

Records may still require attention if they are:

- high-risk
- forbidden
- write-like
- approval-blocked

A record can be safe for review and still require human attention because of policy risk.

## 6. Boundary

V0.9 does not enable:

- n8n runtime integration
- production audit storage
- real provider calls
- credential reads
- file writes from workflow execution
- shell execution
- Git modification
- external write actions
- production autonomous execution

## 7. Tests

Tests cover:

- single sanitized record report generation
- forbidden / high-risk / write-like highlighting
- report excludes raw prompt / raw provider response / Authorization / Bearer / key-like values
- CLI renders Markdown from single JSON input
- CLI renders summary from multi-record JSONL input
