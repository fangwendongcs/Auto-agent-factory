# V0.7 Controlled Execution Boundaries Design

Target:

```text
Design controlled execution boundaries before enabling any real write actions.
```

This document defines the safety and product boundaries required before the project adds any real execution capability beyond read-only provider analysis.

Current baseline:

```text
V0.6c Real Provider Read-only Sandbox + Evaluator Alignment Verified
```

The system can call an OpenAI-compatible provider in a read-only sandbox path, normalize provider output, and evaluate criterion-indexed evidence. Provider output still returns `needs_review` and cannot trigger file writes, shell commands, Git changes, or external write actions.

V0.7 is a design and safety-hardening phase. It is not the phase where real write actions are enabled.

## 1. Goal

Define the control plane required before any future workflow can perform real actions.

V0.7 should answer:

- What kinds of actions are allowed, blocked, or manually gated?
- What approval record is required before an action can run?
- What data must be logged for audit and rollback?
- What happens if an action fails, partially succeeds, or exceeds limits?
- How can the workflow prove that a provider suggestion did not directly become execution?

## 2. Non-goals

V0.7 does not include:

- real file write
- shell execution
- Git modification
- external write API calls
- automatic approval
- production autonomous execution
- Codex execution
- deployment automation
- multi-user permission system
- long-running autonomous loops

## 3. Current safety baseline

Already verified in earlier phases:

- `mock` path works
- `dry-run` path works
- `real-readonly` stub path works
- OpenAI-compatible read-only provider sandbox call works
- provider output remains `needs_review`
- Criteria Checker can evaluate criterion-indexed evidence
- high-risk requests can be blocked before executor dispatch
- invalid payloads can be blocked before executor dispatch
- workflow JSON remains inactive by default in the repository
- real secrets are stored only in n8n credentials or local runtime, not in Git

## 4. Controlled execution principle

Provider output is not authority.

A future execution-capable workflow must treat provider output as a proposal. It may suggest actions, but it must not directly execute them.

The required separation is:

```text
Provider suggestion
→ Result normalization
→ Criteria checking
→ Risk classification
→ Human approval record
→ Execution plan lock
→ Controlled executor
→ Audit log
→ Rollback / recovery path
```

No step may skip human approval for medium or high-risk actions.

## 5. Action classes

Future actions should be classified before execution.

| Action class | Examples | Default policy |
|---|---|---|
| `read_only` | summarize, classify, generate plan, inspect non-secret metadata | allowed after validation |
| `local_draft` | generate draft text, propose patch, create non-applied diff | manual review recommended |
| `repo_write` | modify files, apply patch, update workflow JSON | blocked until explicit approval design exists |
| `shell_command` | run terminal command, install dependency, execute script | blocked by default |
| `git_operation` | commit, push, branch, tag, reset | blocked by default |
| `external_write` | call SaaS mutation API, send email, publish webhook change | blocked by default |
| `deployment` | activate production workflow, deploy service, update infra | blocked by default |

V0.7 should only design these classes. It should not enable blocked classes.

## 6. Approval contract

Before any future write-capable action can run, the workflow should require an approval object.

Recommended shape:

```json
{
  "approval_id": "approval_...",
  "run_id": "gd_...",
  "task_id": "task_...",
  "approved_by": "human",
  "approved_at": "ISO-8601 timestamp",
  "approved_action_classes": ["local_draft"],
  "approved_files": [],
  "approved_external_targets": [],
  "risk_level": "low | medium | high",
  "expires_at": "ISO-8601 timestamp",
  "notes": "Human-readable approval rationale"
}
```

Minimum rules:

- approval must be explicit
- approval must be scoped
- approval must expire
- approval must be tied to `run_id` and `task_id`
- approval must not be inferred from provider confidence
- approval must not be stored as a secret

## 7. Execution plan lock

Before execution, the system should produce a locked plan:

```json
{
  "execution_plan_id": "plan_...",
  "run_id": "gd_...",
  "task_id": "task_...",
  "action_class": "local_draft",
  "intended_actions": [],
  "allowed_targets": [],
  "blocked_targets": [],
  "rollback_plan": [],
  "risk_summary": "...",
  "approval_id": "approval_..."
}
```

The executor should run only the locked plan, not arbitrary provider text.

## 8. Required audit fields

Any future execution attempt should record:

- `run_id`
- `task_id`
- `execution_plan_id`
- `approval_id`
- `action_class`
- `risk_level`
- `started_at`
- `finished_at`
- `status`
- `changed_targets`
- `side_effect_summary`
- `rollback_status`
- `error_context`

Audit logs must avoid storing secrets.

## 9. Failure and rollback expectations

Before enabling real actions, the design must define behavior for:

- approval missing
- approval expired
- action class not allowed
- target not allowed
- timeout
- partial success
- provider response malformed
- executor failure
- rollback failure
- Error Handler triggered

Failure should default to `needs_review` or `failed`, not automatic retry with writes.

## 10. Workflow architecture direction

A future controlled-execution architecture should add a separate execution layer rather than overloading the current read-only Executor.

Suggested future modules:

```text
[GoalDriven] 01 Master
→ [GoalDriven] 02 Agent Task Executor              # read-only planning / provider output
→ [GoalDriven] 03 Criteria Checker                 # evidence evaluation
→ [GoalDriven] 05 Approval Gate                    # approval validation
→ [GoalDriven] 06 Controlled Execution Adapter     # future, disabled by default
→ [GoalDriven] 04 Error Handler
```

Do not add `[GoalDriven] 06 Controlled Execution Adapter` until the approval and audit contracts are accepted.

## 11. V0.7 acceptance criteria

V0.7 is complete when the repository documents:

- action class taxonomy
- approval contract
- execution plan lock contract
- audit fields
- rollback expectations
- failure handling rules
- workflow boundary between provider suggestion and execution
- explicit no-write default policy

V0.7 should also keep all existing checks passing:

```bash
npm test
npm run workflow:validate:all
npm run workflow:dry-run
npm run import:check
```

## 12. Secret and safety requirements

V0.7 must preserve:

- no real API key in Git
- no credential value in workflow JSON
- no Authorization header value in docs or examples
- no `.env` tracking
- no production activation by default
- no hidden execution path

## 13. Next implementation gate

Only after V0.7 is reviewed should the project consider a later phase such as:

```text
V0.8 Approval Gate Prototype
```

Even then, the first implementation should validate approval records and locked plans only. It should still avoid real writes until the user explicitly approves a separate controlled-execution implementation phase.

## 14. Repo-side V0.7 contract implementation

The repository-side V0.7 patch adds a structured approval-decision contract to `[GoalDriven] 01 Master`.

### Implemented fields

`Payload Validator` now emits:

- `action_class`
- `permission_level`
- `forbidden_action_detected`
- `requires_approval_now`
- `approval_decision`

`Blocked Response Builder` now returns the same `approval_decision` contract for invalid, approval-gated, and forbidden requests.

### Implemented decision values

`approval_decision.decision` can be:

- `allow`
- `approved`
- `needs_human_approval`
- `forbidden`
- `invalid_request`

### Implemented enforcement

- high-risk requests without approval are blocked before task initialization
- `repo_write` requires explicit human approval
- `shell_command`, `git_operation`, `external_write`, and `deployment` are forbidden before controlled execution adapters exist
- forbidden action classes are rejected before executor dispatch

### Runtime verification result

Result: **PASS**

The repository-side contract was synced to n8n and verified through the Production Webhook with:

```text
examples/manual-test-payloads/10-v07-read-only-approval-decision.json
examples/manual-test-payloads/11-v07-high-risk-approval-required.json
examples/manual-test-payloads/12-v07-forbidden-shell-command.json
```

Observed runtime result:

- read-only allow: PASS
  - `action_class = read_only`
  - `permission_level = read_only`
  - `approval_decision.decision = allow`
  - `approval_decision.blocked = false`
  - `forbidden_action_detected = false`
  - `status = completed`
- high-risk repo-write approval gate: PASS
  - `status = needs_human_approval`
  - `action_class = repo_write`
  - `permission_level = write_action`
  - `approval_decision.decision = needs_human_approval`
  - `approval_decision.blocked = true`
  - no `agent_result` returned
  - blocked before Executor
- forbidden shell command rejection: PASS
  - `status = forbidden_request`
  - `action_class = shell_command`
  - `permission_level = forbidden`
  - `forbidden_action_detected = true`
  - `approval_decision.decision = forbidden`
  - `approval_decision.blocked = true`
  - rejected before Executor

`criteria_met = false` in the read-only mock regression is not an approval-gate failure; it reflects expected mock executor output for one criterion.

Do not proceed to any write-capable adapter just because V0.7 passed. V0.7 verifies boundaries, not write execution.
