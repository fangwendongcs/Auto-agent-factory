# V0.8b Sanitized Audit Log Prototype

Target:

```text
Create a repo-side sanitized audit record contract and mock sanitizer before connecting any database or write-capable workflow storage.
```

This phase keeps the project in no-write mode. It does not enable n8n file writes, shell execution, Git modification, external write APIs, or production autonomous execution.

## 1. Goal

V0.8b defines a safe audit record prototype that can represent a workflow run without storing sensitive data.

The record should capture:

- sanitized run record
- approval decision
- provider mode / provider status
- timestamps
- risk level
- action class
- permission level
- no-write safety flags
- redaction metadata

The record must not capture:

- full prompt text
- raw provider request messages
- raw provider full response
- API keys
- Authorization headers
- credential values
- cookies
- passwords
- secrets

## 2. Implementation scope

Repo-side artifacts added in this phase:

```text
src/schema/audit-record.schema.json
src/utils/sanitizeAuditRecord.js
examples/sample_sanitized_run_record.json
tests/audit-log.test.js
```

These are local contracts and tests only. They do not persist records to disk at runtime and do not connect to a database.

## 3. Sanitized run record contract

The schema is defined in:

```text
src/schema/audit-record.schema.json
```

Required fields include:

- `audit_record_version`
- `run_id`
- `task_id`
- `record_type`
- `created_at`
- `status`
- `goal_summary`
- `criteria_count`
- `risk_level`
- `action_class`
- `permission_level`
- `provider`
- `approval_decision`
- `timestamps`
- `storage`
- `safety`
- `redaction`

The contract intentionally stores `goal_summary` rather than the full prompt or raw provider request.

## 4. Mock sanitizer behavior

The local utility is defined in:

```text
src/utils/sanitizeAuditRecord.js
```

It creates a sanitized run record from a workflow-like payload and enforces these defaults:

- `storage.mode = mock_repo_contract`
- `storage.persisted = false`
- `safety.no_write_default = true`
- `safety.side_effects_enabled = false`
- `safety.raw_prompt_stored = false`
- `safety.raw_provider_response_stored = false`
- `safety.secrets_stored = false`

It summarizes provider metadata only:

- provider mode
- provider name
- provider model
- provider status

It summarizes approval state only:

- decision
- blocked
- approved
- requires human approval
- reason summary

## 5. Redaction policy

The audit prototype excludes or refuses to copy sensitive raw fields such as:

- `prompt`
- `full_prompt`
- `raw_prompt`
- `messages`
- `provider_request`
- `provider_response_raw`
- `provider_response_content`
- `raw_response`
- `authorization`
- `api_key`
- `token`
- `secret`
- `password`
- `credential`
- `cookie`

The redaction metadata records which field classes are excluded, but it does not store their values.

## 6. Approval semantics

A forbidden action cannot become approved by payload assertion.

If a source payload claims approval for a forbidden action, the sanitized audit record must still produce:

```text
approval_decision.decision = forbidden
approval_decision.approved = false
approval_decision.blocked = true
```

## 7. No-write boundary

V0.8b does not add workflow persistence.

It does not:

- write audit files from n8n
- connect SQLite
- connect external database
- create n8n credentials
- send logs to third-party services
- enable shell, file, Git, or external write actions

## 8. Validation

Local tests cover:

- sample sanitized audit record matches schema
- sanitizer keeps approval and provider summaries only
- sanitizer does not retain sensitive test values from prompt-like or raw response fields
- forbidden audit records remain unapproved even if the source claims approval

Run:

```bash
npm test
npm run workflow:validate:all
npm run workflow:dry-run
npm run import:check
```

## 9. Next gate

After V0.8b is reviewed, the next safe phase can be:

```text
V0.9 sanitized audit log storage design
```

V0.9 should still avoid production execution. If storage is implemented, it should begin with a controlled, local, sanitized, read-only run-record path and an explicit decision about where records are stored.
