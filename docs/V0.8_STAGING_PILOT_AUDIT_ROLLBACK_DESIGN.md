# V0.8 Staging-style Pilot / Audit Logging / Rollback Design

Target:

```text
Design staging-style pilot, audit logging, approval records, persistent run records, failure recovery traces, rollback expectations, and no-write defaults before production.
```

Current baseline:

```text
V0.7 Human-in-the-loop Controlled Execution Boundary Verified
```

V0.8 is a design and readiness phase. It does not enable production autonomous execution, file writes, shell commands, Git modifications, external write actions, or deployment actions.

## 1. Goal

Define how the workflow should be observed, audited, reviewed, and recovered before any future controlled execution capability is considered.

V0.8 focuses on:

- audit logs
- approval records
- persistent run records
- failed execution recovery traces
- rollback expectations
- staging-style pilot checklist
- no-write default policy

The main design question is:

```text
If the system proposes or eventually performs an action, how do we prove who approved it, what it intended to do, what actually happened, and how it can be recovered?
```

## 2. Non-goals

V0.8 does not include:

- real file write
- shell execution
- Git modification
- external write API calls
- deployment or production activation
- automatic approval
- production autonomous execution
- multi-user permission implementation
- long-running autonomous loops
- hidden background jobs

## 3. Current safety baseline

Already verified:

- `mock` path works
- `dry-run` path works
- `real-readonly` stub path works
- OpenAI-compatible read-only provider sandbox path works
- criterion-indexed evidence can produce exact Criteria Checker matches
- high-risk `repo_write` is blocked without explicit approval
- forbidden `shell_command` is rejected before Executor dispatch
- forbidden actions force `approval_decision.approved = false`
- repository workflow JSON remains inactive by default
- real secrets are not stored in Git

## 4. No-write default policy

V0.8 keeps the project in no-write mode.

Default policy:

| Capability | V0.8 default |
|---|---|
| read-only provider analysis | allowed after validation |
| mock execution | allowed |
| dry-run planning | allowed |
| draft-only output | allowed as review material |
| repository writes | blocked |
| file writes | blocked |
| shell commands | blocked |
| Git operations | blocked |
| external write APIs | blocked |
| deployment / activation | blocked |
| automatic approval | blocked |

Provider output remains proposal material. It is not execution authority.

## 5. Persistent run record contract

A future run-history store should persist a sanitized run record.

Recommended shape:

```json
{
  "run_id": "gd_...",
  "created_at": "ISO-8601 timestamp",
  "updated_at": "ISO-8601 timestamp",
  "status": "validated | blocked | needs_human_approval | needs_review | completed | failed",
  "goal_summary": "sanitized short summary",
  "criteria_count": 3,
  "risk_level": "low | medium | high",
  "action_class": "read_only | local_draft | repo_write | shell_command | git_operation | external_write | deployment",
  "permission_level": "read_only | draft | write_action | forbidden",
  "provider_mode": "mock | dry-run | real-readonly",
  "provider_execution": "stub | provider | none",
  "max_iterations": 1,
  "timeout_minutes": 5,
  "approval_decision": {},
  "task_ids": [],
  "final_next_action": "stop | ask_human | revise | rollback | blocked",
  "safe_to_retry": false
}
```

Rules:

- store summaries, not secrets
- avoid full sensitive payload capture by default
- keep `run_id` and `task_id` stable for traceability
- record blocked decisions, not only successful paths
- record no-write status explicitly

## 6. Approval record contract

Approval records should be persisted separately from provider output.

Recommended shape:

```json
{
  "approval_id": "approval_...",
  "run_id": "gd_...",
  "task_id": "task_...",
  "decision": "allow | approved | needs_human_approval | forbidden | rejected",
  "approved_by": "human",
  "approved_at": "ISO-8601 timestamp",
  "expires_at": "ISO-8601 timestamp",
  "risk_level": "low | medium | high",
  "action_class": "repo_write",
  "permission_level": "write_action",
  "approved_scope": {
    "allowed_action_classes": [],
    "allowed_targets": [],
    "blocked_targets": [],
    "max_iterations": 1,
    "timeout_minutes": 5
  },
  "reason": "Human-readable rationale",
  "source": "n8n-ui | manual-review | future-approval-ui"
}
```

Rules:

- approval must be explicit
- approval must be scoped
- approval must expire
- approval must be tied to `run_id` and `task_id`
- approval must not be inferred from provider confidence
- approval must not override forbidden actions
- approval records must not contain secrets

## 7. Audit event contract

Audit events should record important state transitions.

Recommended shape:

```json
{
  "event_id": "evt_...",
  "run_id": "gd_...",
  "task_id": "task_...",
  "timestamp": "ISO-8601 timestamp",
  "event_type": "payload_validated | approval_required | forbidden_rejected | executor_dispatched | provider_response_received | criteria_checked | final_report_generated | error_captured | rollback_recommended",
  "node_name": "Payload Validator",
  "status_before": "validated",
  "status_after": "needs_human_approval",
  "summary": "short sanitized event summary",
  "risk_level": "high",
  "action_class": "repo_write",
  "approval_decision": {},
  "error_context": null
}
```

Rules:

- never log API keys
- never log Authorization headers
- never log credential values
- keep raw provider output optional and sanitized
- record blocked paths as first-class audit events

## 8. Failure recovery trace

A failed execution should produce a recovery trace, not just an error message.

Recommended shape:

```json
{
  "recovery_id": "recovery_...",
  "run_id": "gd_...",
  "task_id": "task_...",
  "failed_workflow": "[GoalDriven] 01 Master",
  "failed_node": "...",
  "failure_type": "validation | approval | provider | criteria_checker | error_handler | unknown",
  "error_message": "sanitized message",
  "last_safe_state": "...",
  "side_effects_detected": false,
  "recommended_action": "retry_readonly | ask_human | inspect_execution | rollback_not_required",
  "rollback_required": false,
  "rollback_plan": []
}
```

Rules:

- failure defaults to human review
- retry must not create write side effects
- if side effects are unknown, mark as `ask_human`
- Error Handler should remain the fallback for unexpected runtime failures

## 9. Rollback expectation

Because V0.8 remains no-write, most rollback actions should be documented as:

```text
rollback_required = false
```

For future write-capable phases, rollback expectations must be defined before execution:

- what target may change
- how to restore the previous state
- what evidence proves rollback succeeded
- what happens if rollback fails
- who reviews the rollback result

No future write action should run without a rollback expectation or an explicit statement that rollback is not possible and requires human acceptance.

## 10. Staging-style pilot checklist

Before any future execution-capable pilot, complete this checklist:

- [ ] Git working tree is clean
- [ ] workflow JSON is validated locally
- [ ] workflow exports are inactive by default in Git
- [ ] n8n runtime workflow is backed up
- [ ] Error Handler is configured
- [ ] approval boundary is verified
- [ ] forbidden action rejection is verified
- [ ] audit record destination is selected
- [ ] secrets are stored only in n8n credentials or ignored local runtime config
- [ ] provider output remains `needs_review`
- [ ] no-write default policy is confirmed
- [ ] rollback expectation is documented
- [ ] one manual reviewer is identified
- [ ] one small staging payload is selected
- [ ] no production webhook is exposed publicly without authentication

## 11. Possible storage options

V0.8 does not choose a final persistence backend. Acceptable future options include:

| Option | Use case | Notes |
|---|---|---|
| local JSONL file | local development only | must stay out of Git if it can contain sensitive data |
| SQLite | simple local persistence | good for deterministic pilot logs |
| n8n data store | n8n-native state | check availability by n8n version |
| GitHub Issues | human-readable review queue | avoid secrets and raw sensitive payloads |
| external database | future production-like setup | requires credential and access control design |

No persistence option should store secrets or Authorization headers.

## 12. V0.8 acceptance criteria

V0.8 is complete when the repository documents:

- no-write default policy
- persistent run record contract
- approval record contract
- audit event contract
- failure recovery trace contract
- rollback expectations
- staging-style pilot checklist
- storage options and secret boundaries

V0.8 should also keep local checks passing:

```bash
npm test
npm run workflow:validate:all
npm run workflow:dry-run
npm run import:check
```

## 13. Next implementation gate

Only after V0.8 is reviewed should the project consider:

```text
V0.9 Audit Log Prototype
```

A safe V0.9 should persist sanitized read-only run records only. It should still avoid file writes, shell execution, Git modification, external write APIs, and production autonomous execution unless a separate explicit phase approves a controlled storage mechanism.
