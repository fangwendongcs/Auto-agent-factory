# V0.8c Dev-only Audit Storage Plan

Target:

```text
Plan how sanitized audit records can be persisted safely in development without enabling production write capabilities.
```

Current baseline:

```text
V0.8b sanitized audit log prototype contract
```

V0.8c is a storage planning phase. It does not modify n8n runtime, create credentials, call providers, or enable workflow write actions.

## 1. Goal

Define a safe persistence approach for sanitized audit records.

The storage plan must support:

- sanitized run record persistence
- approval decision persistence
- provider mode / status / timestamps
- failure recovery trace references
- rollback expectation references
- local inspection during staging-style pilots

The storage plan must not store:

- raw prompts
- provider request messages
- raw provider full responses
- API keys
- Authorization headers
- credential values
- cookies
- passwords
- secrets

## 2. Non-goals

V0.8c does not include:

- production database setup
- n8n runtime changes
- n8n file writes
- shell execution
- Git modification
- external write API calls
- provider calls
- automatic approval
- production autonomous execution

## 3. Storage options comparison

| Option | Strengths | Risks / limitations | Fit for next phase |
|---|---|---|---|
| Dev-only local JSONL | Transparent, append-friendly, easy to inspect, simple test fixture, no database dependency | File write must be explicitly dev-only and ignored by Git; not suitable for production access control | Recommended first prototype |
| SQLite | Structured queries, single local file, better for later reporting | Adds schema migration concerns; still a local write; easier to overbuild too early | Good later if JSONL becomes limiting |
| n8n execution reference only | No new storage writes; uses existing n8n execution history | Execution data may be pruned; harder to query; may include more raw context than desired | Useful as supplemental reference, not primary contract |
| Manual export | Safest operationally; user controls export timing | Not automated; inconsistent; poor for regression testing | Good for one-off review, not a prototype store |

## 4. Recommended approach

Recommended next implementation:

```text
Dev-only local JSONL adapter, disabled by default.
```

Recommended path if implemented later:

```text
.local-audit/dev-audit-log.jsonl
```

Rules:

- directory must be ignored by Git
- storage must be disabled unless explicitly enabled
- only sanitized audit records may be written
- each line must validate against `src/schema/audit-record.schema.json`
- writer must reject records containing denylisted fields or secret-like values
- no n8n workflow should write this file in V0.8c
- no production usage claim

## 5. Dev-only enablement strategy

Future implementation should require an explicit local flag such as:

```text
AUDIT_STORAGE_MODE=dev-jsonl
AUDIT_STORAGE_ENABLED=true
AUDIT_STORAGE_PATH=.local-audit/dev-audit-log.jsonl
```

Default behavior:

```text
AUDIT_STORAGE_ENABLED=false
```

If the flag is missing, storage should not write anything.

Recommended safeguards:

- fail closed when storage path is outside the approved local audit directory
- reject absolute paths by default
- reject paths containing `..`
- create directories only under `.local-audit/`
- never write to `docs/`, `examples/`, `workflows/`, `src/`, or `tests/`
- never write during `npm test` unless a temporary directory is injected by the test

## 6. Field allowlist

Only these top-level fields should be persisted:

- `audit_record_version`
- `run_id`
- `task_id`
- `record_type`
- `created_at`
- `status`
- `goal_summary`
- `criteria_count`
- `risk_level`
- `action_class`
- `permission_level`
- `provider`
- `approval_decision`
- `timestamps`
- `storage`
- `safety`
- `redaction`

Allowed nested provider fields:

- `provider.mode`
- `provider.name`
- `provider.model`
- `provider.status`

Allowed nested approval fields:

- `approval_decision.decision`
- `approval_decision.blocked`
- `approval_decision.approved`
- `approval_decision.requires_human_approval`
- `approval_decision.reason_summary`

## 7. Field denylist

These fields must never be persisted:

- `prompt`
- `full_prompt`
- `raw_prompt`
- `messages`
- `provider_request`
- `provider_response_raw`
- `provider_response_content`
- `raw_response`
- `authorization`
- `Authorization`
- `api_key`
- `OPENAI_API_KEY`
- `N8N_API_KEY`
- `token`
- `secret`
- `password`
- `credential`
- `cookie`
- `headers`
- `body` when it contains raw provider payload

Denylist applies recursively.

## 8. Secret validation requirements

Before writing any future record, the storage adapter should verify:

- schema validation passes
- record contains no denylisted field names outside `redaction.excluded_fields`
- serialized record does not contain `Bearer `
- serialized record does not contain `sk-`
- serialized record does not contain known token prefixes such as `ghp_` or `xoxb-`
- serialized record does not contain `Authorization`
- `safety.secrets_stored = false`
- `safety.raw_prompt_stored = false`
- `safety.raw_provider_response_stored = false`

If any check fails, the adapter must refuse to write and return a structured local error.

## 9. Shutdown / rollback / cleanup

Dev-only storage must be easy to disable and clean up.

Disable:

```text
AUDIT_STORAGE_ENABLED=false
```

Cleanup:

```text
remove .local-audit/ only after confirming no needed local evidence remains
```

Rollback expectations:

- delete or archive local dev audit file
- keep Git unchanged
- do not remove n8n executions
- do not change workflow JSON
- rerun local checks after cleanup

The cleanup operation should remain manual unless the user explicitly asks for a cleanup script.

## 10. Required .gitignore rule

Before implementing dev JSONL storage, ensure `.gitignore` includes:

```text
.local-audit/
*.jsonl
```

If `*.jsonl` is too broad for future project needs, use the narrower rule:

```text
.local-audit/*.jsonl
```

Recommended first choice:

```text
.local-audit/
```

## 11. Future implementation shape

A minimal future implementation can add:

```text
src/utils/auditStorage.js
scripts/write-audit-record.mjs
tests/audit-storage.test.js
```

The first implementation should:

- accept only already-sanitized records
- validate schema before write
- validate secret safety before write
- append JSONL only when explicitly enabled
- support test injection of a temporary output path
- never run as part of n8n runtime by default

## 12. Acceptance criteria for V0.8c

V0.8c is complete when:

- storage options are compared
- dev-only JSONL is selected as the recommended first prototype
- field allowlist is documented
- field denylist is documented
- secret validation requirements are documented
- enable / disable / cleanup strategy is documented
- production usage is explicitly out of scope
- local checks pass

## 13. Next recommendation

After this plan is reviewed, the next safe implementation phase can be:

```text
V0.8d dev-only JSONL audit storage prototype
```

V0.8d should remain disabled by default and should write only to a local ignored path when explicitly enabled.
