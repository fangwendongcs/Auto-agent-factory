# V0.4c Real-readonly Implementation Plan

This document defines the minimal implementation plan for adding an OpenAI-compatible real-readonly provider path to `[GoalDriven] 02 Agent Task Executor`.

This is a plan, not an implementation. It does not connect a real API, create credentials, or modify workflow JSON.

## 1. V0.4 implementation goal

V0.4 adds a controlled real provider path that can perform read-only analysis and return a normalized `agent_result`.

The first provider interface is:

```text
OpenAI-compatible HTTP interface
```

This is not a vendor lock-in decision. The same interface can later point to:

- OpenRouter-compatible endpoint
- OpenAI-compatible endpoint
- local OpenAI-compatible HTTP agent
- another compatible gateway

### What it can do

- call a read-only provider endpoint
- generate `summary`
- generate `intended_actions`
- generate `evidence`
- generate `risk_summary`
- return provider metadata
- return `status = needs_review`
- preserve the existing `agent_result` contract

### What it cannot do

- write files
- execute shell commands
- modify Git
- call external write APIs
- approve its own output
- bypass human approval
- start autonomous production execution
- store or print real secrets

## 2. Minimal Executor workflow change

The first implementation should modify only:

```text
workflows/agent_task_executor.workflow.json
```

Do not modify:

```text
workflows/goal_driven_master.workflow.json
workflows/criteria_checker.workflow.json
workflows/error_handler.workflow.json
```

Current verified paths must remain intact:

```text
mock          → Mock Agent Adapter
dry-run       → Dry-run Provider Adapter
real-readonly → Real-readonly Provider Adapter
```

### Proposed node structure

The smallest safe evolution is to split the current real-readonly path internally while keeping the existing mode router behavior:

```text
When Executed by Another Workflow
→ Task Validator
→ Prompt Builder
→ Mode Router
   ├─ Mock Agent Adapter
   ├─ Dry-run Provider Adapter
   └─ Real-readonly Provider Adapter
        ├─ Real-readonly Provider Selector
        ├─ Real-readonly Safety Check
        ├─ OpenAI-compatible HTTP Request
        ├─ Provider Response Normalizer
        └─ Real-readonly Stub Fallback
→ Result Normalizer
→ Execution Logger
```

Recommended behavior:

- If `context.provider_execution = "stub"` or no real provider is configured, use the existing real-readonly stub.
- If `context.provider_execution = "provider"` and credential/config checks pass, call the OpenAI-compatible HTTP provider.
- If the provider call fails or response normalization fails, return a structured failure `agent_result` rather than allowing an unhandled crash whenever possible.

## 3. Provider request / response contract

### Endpoint configuration

Use placeholders only in workflow JSON and documentation:

```text
OPENAI_COMPATIBLE_BASE_URL
OPENAI_COMPATIBLE_MODEL
```

Recommended n8n credential name:

```text
goald-openai-compatible-readonly
```

Do not store the API key in:

- workflow JSON
- README
- docs
- examples
- logs
- Git

### HTTP request shape

The OpenAI-compatible request should be constrained to read-only JSON output:

```json
{
  "model": "={{ $json.provider_config.model }}",
  "messages": [
    {
      "role": "system",
      "content": "You are a read-only planning and evidence generation adapter. Do not execute commands, write files, modify Git, call external write APIs, or claim that work was executed. Return JSON only."
    },
    {
      "role": "user",
      "content": {
        "mode": "real-readonly",
        "goal": "...",
        "criteria": ["..."],
        "iteration": 1,
        "constraints": [
          "Do not execute commands.",
          "Do not write files.",
          "Do not modify Git.",
          "Do not call external write APIs.",
          "Return structured JSON only."
        ],
        "expected_output": {
          "summary": "string",
          "intended_actions": ["string"],
          "evidence": [
            {
              "criterion": "string",
              "status": "pass | fail | unknown",
              "detail": "string"
            }
          ],
          "risk_summary": "string"
        }
      }
    }
  ],
  "temperature": 0.2,
  "response_format": {
    "type": "json_object"
  }
}
```

If a provider does not support `response_format`, the normalizer must still validate the returned content and fail safely when it is not JSON.

### Expected provider output

The provider should return JSON content equivalent to:

```json
{
  "summary": "...",
  "intended_actions": [],
  "evidence": [
    {
      "criterion": "...",
      "status": "pass | fail | unknown",
      "detail": "..."
    }
  ],
  "risk_summary": "..."
}
```

The provider output is not trusted until normalized.

## 4. Response normalization

`Provider Response Normalizer` must convert provider-specific output into the existing `agent_result` contract.

Required normalized fields:

```json
{
  "run_id": "...",
  "task_id": "...",
  "status": "needs_review",
  "summary": "...",
  "artifacts": [],
  "known_issues": [],
  "evidence": [],
  "intended_actions": [],
  "risk_summary": "...",
  "provider": {
    "mode": "real-readonly",
    "name": "openai-compatible-readonly",
    "model": "...",
    "request_id": "...",
    "latency_ms": 0
  },
  "safety": {
    "risk_level": "low | medium | high",
    "requires_human_approval": true,
    "approved": false,
    "blocked_reason": null
  }
}
```

Rules:

- `status` must remain `needs_review` on provider success.
- `artifacts` must remain an array and should default to `[]`.
- `known_issues` must remain an array and should default to `[]`.
- `evidence` must be an array.
- each evidence item must include `criterion`, `status`, and `detail`.
- unknown or unsupported evidence status must become `unknown`.
- missing `summary` should become a safe fallback summary.
- missing `risk_summary` should become `"Risk summary was not provided by the provider."`
- no raw secret or full authorization header may be included in logs.

## 5. Failure fallback

Provider failures should return a structured failure result whenever possible.

### Endpoint not configured

Return:

```json
{
  "status": "needs_review",
  "summary": "Real-readonly provider endpoint is not configured.",
  "known_issues": ["provider_endpoint_missing"],
  "provider": {
    "mode": "real-readonly",
    "name": "openai-compatible-readonly",
    "model": null,
    "request_id": null,
    "latency_ms": 0
  },
  "safety": {
    "requires_human_approval": true,
    "approved": false,
    "blocked_reason": "provider_endpoint_missing"
  }
}
```

### Credential missing

Return:

```json
{
  "status": "needs_review",
  "summary": "Real-readonly provider credential is missing.",
  "known_issues": ["provider_credential_missing"],
  "safety": {
    "requires_human_approval": true,
    "approved": false,
    "blocked_reason": "provider_credential_missing"
  }
}
```

### Timeout

Return `status = failed` with:

```text
blocked_reason = provider_timeout
```

### HTTP 401 / 403

Return `status = failed` with:

```text
blocked_reason = provider_auth_error
```

Do not print the credential or token.

### HTTP 429

Return `status = failed` with:

```text
blocked_reason = provider_rate_limited
```

### HTTP 5xx

Return `status = failed` with:

```text
blocked_reason = provider_unavailable
```

### Invalid JSON

Return `status = failed` with:

```text
blocked_reason = provider_invalid_json
```

### Required fields missing

Return `status = failed` with:

```text
blocked_reason = provider_contract_invalid
```

If the n8n HTTP node itself throws before fallback logic can run, `[GoalDriven] 04 Error Handler` should capture the failed execution.

## 6. Safety boundaries

V0.4 must preserve these boundaries:

- no file write
- no shell execution
- no Git modification
- no external write action
- no real secret in repo
- no API key in workflow JSON
- no API key in docs
- no API key in logs
- high-risk requests still require human approval
- provider output remains `needs_review`
- provider output does not become execution automatically

## 8. V0.4 acceptance criteria

V0.4 is acceptable only if:

- existing mock path is not broken
- existing dry-run path is not broken
- existing real-readonly stub remains available or has a safe fallback
- OpenAI-compatible provider path succeeds in read-only mode
- provider path returns normalized `agent_result`
- Criteria Checker does not need to change
- Master does not need to change
- Error Handler does not need to change
- `workflow:validate:all` remains `0 warning / 0 error`
- no real secret is committed
- n8n execution path can be inspected and verified
- no file write / shell execution / Git modification / external write action occurs

## 9. Out of scope

V0.4 does not include:

- real file write
- shell execution
- Git commit / push
- production autonomous execution
- multi-provider routing
- long-term memory
- RAG integration
- cost optimization
- dashboard UI
- automatic approval
- tool-calling execution
