# PRODUCT_SPEC

## 1. 产品名称

**Goal-Driven Agent Workflow with n8n**

## 2. 产品定位

这是一个基于 n8n 的可复用 Agent 工作流框架。  
它不是单次自动化脚本，也不是某个 Agent 的专属外挂，而是一套围绕 `goal + criteria` 运行的任务编排模板。

系统接收用户目标后，不直接宣称“完成”，而是通过持续迭代来逼近结果：

1. 初始化运行记录
2. 调用执行 Agent
3. 检查本轮产出是否满足 criteria
4. 未达标时生成下一轮指令
5. 达标、超时或达到最大迭代次数后停止并生成报告

## 3. 目标用户

### 主要用户

- 想把“目标 → 拆解 → 执行 → 检查 → 重试”自动化的人
- 希望在 n8n 中复用 Agent 闭环的人
- 需要保留人工审核、日志和可回滚性的个人开发者或小团队

### 典型场景

- 自动生成研究报告
- 自动迭代文档或方案
- 自动生成并检查工作流设计
- 将来接入 Codex、GitHub Issue、HTTP Agent 或其他执行器

## 4. 核心问题

现有 Agent 自动化常见问题：

- 只有任务，没有可验证的成功标准
- 执行一次后就停止，无法自动修正
- 没有最大迭代次数，容易无限消耗时间和 token
- 没有错误处理和日志，失败后无法恢复
- 没有人类审核节点，高风险任务容易失控

## 5. 产品目标

### MVP 必须实现

- 接收 `goal` 和 `criteria`
- 生成 `run_id` 和 `task_id`
- 调用执行 Agent
- 逐项检查 criteria
- 未达标时生成下一轮指令
- 达标后停止并生成最终报告
- 达到 `max_iterations` 或超时后安全停止
- 具备错误处理 workflow
- 具备人工审核入口
- 支持 mock 模式

### MVP 不包含

- 真正的多租户权限系统
- 复杂前端控制台
- 高级可观测平台
- 长期记忆 / RAG 全量实现
- 自动生产部署
- 直接依赖某一个固定的 LLM 或 Codex API

## 6. 用户旅程

```text
用户输入 Goal + Criteria
        ↓
系统创建 run_id / task_id
        ↓
Executor 产出本轮结果
        ↓
Checker 逐项核验 criteria
        ↓
达标：生成最终报告
未达标：生成下一轮任务
失败：进入 Error Handler
```

## 7. 核心对象

### Goal

- 用户想达成什么
- 如何判断达成
- 最多允许多少轮
- 最长允许运行多久

### Task

- 本轮要执行什么
- 由哪个 Agent 执行
- 当前状态是什么

### Result

- 本轮执行产物
- 已知问题
- 证据
- 下一步建议

## 8. 关键设计原则

1. **先定义标准，再允许执行**
2. **每一轮都要可解释**
3. **Agent 结果必须结构化**
4. **系统必须有停止条件**
5. **高风险动作必须可人工中断**
6. **真实密钥绝不进入仓库**
7. **正式 workflow 必须可导入、可审查、可回滚**

## 9. 与旧原型的关系

旧版 `Codex Planner Reviewer Workflow` 证明了：

- workflow 可以作为代码管理
- 可以在 workflow 中加入安全守卫与人工闸门
- 可以把 Agent 之间的输入输出格式化

新项目会保留这些经验，但把范围从“围绕 Codex 的单流程原型”扩展为“围绕 Goal-Driven 闭环的通用模板”。
