# V0.6 Evaluator Quality / Reliability / Safety Hardening Plan

This document defines the evaluator-quality phase after V0.5c and records the V0.6b / V0.6c direction.

Current baseline before V0.6:

```text
V0.5c Real Provider Read-only Sandbox + Checker Alignment Verified
```

Current verified status after V0.6c:

```text
V0.6c Real Provider Read-only Sandbox + Evaluator Alignment Verified
```

The system can call an OpenAI-compatible provider through the read-only sandbox path and normalize provider output into `agent_result`. V0.6b added criterion-indexed provider evidence, and V0.6c runtime verification confirmed exact Criteria Checker matches for the sandbox payload. Further evaluator quality work should focus on ambiguous or weak evidence, not on enabling execution.

V0.6 is still **read-only first**. It is not production autonomous execution.

## 1. Goal

Improve the reliability and interpretability of Criteria Checker decisions when evaluating provider-generated output against user-provided criteria.

V0.6 focuses on:

- better mapping between `criteria[]` and `agent_result.evidence[]`
- clearer scoring and failure reasons
- stricter evidence quality expectations
- safer handling of weak or generic provider output
- regression coverage for mock, dry-run, real-readonly stub, and real provider sandbox paths

The goal is not to force `criteria_met = true`. The goal is to make pass/fail/unknown decisions more explainable and less dependent on fragile index fallback.

## 2. Non-goals

V0.6 does not enable:

- file write
- shell execution
- Git modification
- external write action
- automatic approval
- production autonomous execution
- long-running autonomous loops
- Codex execution
- multi-provider routing
- cost optimization
- persistent run history

Provider success must still return `needs_review` unless a future explicitly reviewed phase changes that boundary.

## 3. Current known issue

V0.5c confirmed that Criteria Checker can read provider evidence, provider context, and normalized `agent_result` fields.

Remaining issue:

```text
criteria_met may remain false because provider evidence and input criteria can be semantically misaligned.
```

Examples:

- Criteria may say `Output must include intended_actions`.
- Provider evidence may label the item as `planning_actions`, `steps`, or a broad natural-language paragraph.
- The checker may fall back to `index_fallback`, which proves that evidence exists but does not strongly prove semantic satisfaction.

This is a quality issue, not an execution failure.

## 4. Proposed evaluator contract hardening

Future provider evidence should prefer a more explicit shape:

```json
{
  "criterion": "Output must include intended_actions",
  "criterion_id": "c1",
  "status": "pass | fail | unknown",
  "detail": "Specific evidence explaining how this criterion is or is not satisfied.",
  "confidence": 0.0,
  "supports_fields": ["intended_actions"],
  "limitations": []
}
```

Recommended additions:

- `criterion_id`: stable ID generated by the workflow or checker
- `confidence`: numeric confidence from `0` to `1`
- `supports_fields`: normalized fields used as evidence
- `limitations`: reasons why the evidence may be incomplete

The existing `agent_result.evidence[]` contract should remain backward-compatible. New fields should be optional at first.

## 5. Checker scoring direction

V0.6 should move from simple evidence presence to weighted evidence quality.

Suggested match strength order:

1. exact criterion ID match
2. exact criterion text match
3. normalized criterion text match
4. explicit supported field match
5. provider context fallback
6. index fallback

Suggested behavior:

- `pass` requires explicit evidence, not just field existence
- `unknown` should not become `pass` automatically
- `index_fallback` should be marked as weak evidence
- vague evidence should produce a clear issue, not a silent pass
- high-risk outputs should never auto-pass into execution

## 6. Provider prompt alignment

The provider prompt should ask the model to produce evidence item-by-item against the original criteria.

The expected provider response should remain JSON-only and read-only:

```json
{
  "summary": "...",
  "intended_actions": [],
  "evidence": [],
  "risk_summary": "..."
}
```

Recommended prompt refinement:

- include numbered criteria with stable IDs
- require one evidence item per criterion
- require `status = pass | fail | unknown`
- require a concise `detail` for each criterion
- require limitations when evidence is incomplete
- forbid claims that real actions were executed
- remind the provider that output is read-only planning

## 7. Safety boundaries

These boundaries stay mandatory throughout V0.6:

- no file write
- no shell execution
- no Git modification
- no external write action
- no automatic approval
- no production autonomous execution
- no real secret in repository files
- high-risk requests remain blocked before executor dispatch unless explicitly approved
- provider output remains review material, not executable authority

## 8. Suggested implementation phases

### V0.6a — Plan and fixtures

- Document evaluator quality requirements.
- Add non-secret sample provider outputs if useful.
- Do not modify workflow logic.

### V0.6b — Provider prompt refinement

- Improve the OpenAI-compatible request builder prompt to request criterion-indexed evidence.
- Keep provider path read-only.
- Preserve existing mock, dry-run, and real-readonly stub paths.

### V0.6c — Checker scoring refinement

- Improve Criteria Checker scoring and `match_type` semantics.
- Add stronger distinction between exact matches, context fallback, and index fallback.
- Keep output contract backward-compatible.

### V0.6d — Runtime regression

- Verify through Production Webhook:
  - mock regression
  - dry-run regression
  - real-readonly stub regression
  - real provider sandbox regression
  - high-risk approval gate
  - missing goal validation
  - missing criteria validation

## 9. Regression test plan

After any V0.6 workflow patch, run local checks:

```bash
npm test
npm run workflow:validate:all
npm run workflow:dry-run
npm run import:check
```

Runtime regression should verify:

- `mock` still routes to Mock Agent Adapter
- `dry-run` still routes to Dry-run Provider Adapter
- `real-readonly` stub still routes to Real-readonly Provider Adapter
- provider sandbox path remains read-only
- provider success still returns `needs_review`
- Criteria Checker emits explainable `checks[]`
- `match_type` is visible
- weak matches are not silently treated as strong passes
- high-risk requests are blocked before executor dispatch
- invalid payloads are blocked before executor dispatch
- no secret is exposed in workflow output

## 10. Acceptance criteria

V0.6 can be considered complete when:

- provider evidence maps more clearly to criteria
- checker output explains why each criterion passed, failed, or remained unknown
- index fallback is no longer the primary evidence path for normal provider outputs
- weak evidence is explicitly marked as weak
- no automatic approval is introduced
- existing paths remain working
- local checks pass
- workflow validation remains 0 warning / 0 error
- secret audit passes

## 11. Out of scope for V0.6

Do not include:

- real write execution
- shell execution
- Git commit / push automation
- file generation by the workflow
- production deployment automation
- human approval UI
- long-term run history
- multi-provider routing
- RAG integration
