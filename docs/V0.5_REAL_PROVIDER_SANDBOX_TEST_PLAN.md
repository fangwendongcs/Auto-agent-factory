# V0.5 Real Provider Sandbox Test Plan

This document defines the sandbox test plan for the first real provider validation.

V0.5 is still **read-only first**. It is not production autonomous execution.

## 1. Goal

Validate that the OpenAI-compatible real-readonly provider path can call a real sandbox provider and return a normalized `agent_result`.

The provider may generate:

- `summary`
- `intended_actions`
- `evidence`
- `risk_summary`
- provider metadata

The provider must return `status = needs_review`.

## 2. Non-goals

V0.5 does not include:

- file write
- shell execution
- Git modification
- external write action
- production autonomous execution
- automatic approval
- multi-provider routing
- long-running agent loops
- Codex execution
- deployment automation

## 3. Preconditions

Before running V0.5 sandbox tests:

- V0.4e runtime regression remains verified
- Git working tree is clean
- local checks pass:

  ```bash
  npm test
  npm run workflow:validate:all
  npm run workflow:dry-run
  npm run import:check
  ```

- `[GoalDriven] 02 Agent Task Executor` is backed up from n8n UI
- the current n8n runtime points Master to the correct 02 Executor workflow
- Error Handler remains configured
- high-risk approval gate remains verified
- no real API key is stored in the repository

Manual setup checklist:

```text
docs/V0.5_SANDBOX_MANUAL_SETUP_CHECKLIST.md
```

## 4. Credential setup

Create a sandbox credential in n8n UI only.

Recommended credential name:

```text
goald-openai-compatible-readonly
```

Rules:

- do not write the key into workflow JSON
- do not write the key into README or docs
- do not commit the key
- do not paste the key into logs
- do not expose the key in webhook payloads

The credential should only be used by the read-only provider path.

## 5. Environment / placeholder configuration

Use placeholders or local runtime configuration only.

Allowed placeholder names:

```text
OPENAI_COMPATIBLE_BASE_URL
OPENAI_COMPATIBLE_MODEL
OPENAI_COMPATIBLE_TIMEOUT_MS
```

Recommended sandbox values should be configured in n8n runtime or a local ignored environment file, not committed to Git.

Example non-secret config shape:

```json
{
  "provider_execution": "provider",
  "provider_name": "openai-compatible-readonly",
  "provider_credential_name": "goald-openai-compatible-readonly",
  "provider_credential_ready": true,
  "openai_compatible_base_url": "configured-in-n8n-runtime",
  "openai_compatible_model": "configured-in-n8n-runtime"
}
```

Do not replace placeholder values with real secrets in repository files.

## 6. Provider success path acceptance criteria

The sandbox provider success path passes only if:

- Production Webhook returns `run_id`
- Production Webhook returns `task_id`
- `agent_result.status = needs_review`
- `agent_result.provider.mode = real-readonly`
- `agent_result.provider.name = openai-compatible-readonly`
- `agent_result.summary` exists
- `agent_result.intended_actions` exists
- `agent_result.evidence` exists
- `agent_result.risk_summary` exists
- `agent_result.safety.requires_human_approval = true`
- Criteria Checker can process the normalized evidence without provider-specific logic
- n8n execution path is visible and matches:

  ```text
  Real-readonly Provider Selector
  → Real-readonly Safety Check
  → OpenAI-compatible provider call
  → Provider Response Normalizer
  → Result Normalizer
  → Execution Logger
  ```

- no file write
- no shell execution
- no Git modification
- no external write action

## 7. Provider failure path acceptance criteria

Failure handling must be tested for:

- missing endpoint
- missing credential
- timeout
- HTTP 401 / 403
- HTTP 429
- HTTP 5xx
- invalid JSON
- missing required fields

Each failure path should return a structured safe result where possible:

- `status = needs_review` or `status = failed`
- `provider.mode = real-readonly`
- `known_issues` includes a safe failure reason
- `safety.requires_human_approval = true`
- `safety.blocked_reason` is set
- no secret is printed
- no real side effect occurs

If the n8n node itself throws before normalization, `[GoalDriven] 04 Error Handler` should capture the failed execution.

## 8. Cost, timeout, and rate limits

V0.5 sandbox testing should use a low-cost model and a small payload.

Recommended safeguards:

- `max_iterations = 1`
- `timeout_minutes <= 5`
- provider timeout around `30000ms`
- low token limit
- one request at a time
- no looped provider calls
- manual inspection after each run

Do not run unattended repeated tests.

## 9. Safety boundaries

These boundaries remain mandatory:

- read-only first
- no file write
- no shell execution
- no Git modification
- no external write action
- no production autonomous execution
- no automatic approval
- high-risk requests still require human approval
- provider output remains `needs_review`
- provider output cannot trigger execution

## 10. Secret audit requirements

After sandbox testing:

1. inspect n8n execution output for accidental secret exposure
2. inspect local Git diff
3. run a secret audit across changed files
4. confirm `.env` is not tracked
5. confirm workflow JSON contains only placeholder names or credential references

Expected result:

```text
real secret values found: no
safe to commit from secret perspective: yes
```

## 11. Rollback strategy

Before testing:

1. export `[GoalDriven] 02 Agent Task Executor`
2. save backup as:

   ```text
   agent_task_executor-before-v05-provider-sandbox.json
   ```

3. confirm Master still points to the intended Executor workflow

If sandbox testing fails:

1. restore the backup 02 workflow in n8n UI
2. publish / save the restored workflow
3. rerun mock regression
4. rerun dry-run regression
5. rerun real-readonly stub regression
6. verify high-risk approval gate
7. verify invalid payload validation

Do not patch Master, Checker, or Error Handler unless evidence shows the failure is outside Executor.

## 12. V0.5 completion criteria

V0.5 can be considered complete when:

- provider success path passes once in sandbox
- at least one provider failure path is verified
- mock regression passes
- dry-run regression passes
- real-readonly stub regression passes
- high-risk approval gate passes
- invalid payload validation passes
- local checks pass
- secret audit passes
- validation result is recorded in `docs/VALIDATION_LOG.md`

## 13. Next phase after V0.5

Only after V0.5 passes, the project may enter:

```text
V0.6 Provider Observability and Run Reporting
```

V0.6 should focus on:

- provider latency
- token / cost metadata
- request ID tracking
- run history
- final report quality
- failure classification

It should still avoid file writes, shell execution, Git modification, and production autonomous execution.
